    Polars Integration - Phase 1¶

    Overview¶

    Phase 1 of the Polars integration adds Polars LazyFrame support alongside the existing SQL backend in SLAF. This phase focuses on:

    1. Adding Polars dependencies without breaking existing functionality
    2. Creating PolarsQueryOptimizer that reuses existing query optimization logic
    3. Extending SLAFArray with Polars LazyFrame support
    4. Adding comprehensive testing and benchmarking
    5. Maintaining backward compatibility with existing SQL interface

    Key Components¶

    1. PolarsQueryOptimizer (slaf/core/polars_optimizer.py)¶

    The PolarsQueryOptimizer class adapts the existing SQL query optimizations to Polars expressions:

    • Consecutive ID Detection: Uses BETWEEN instead of IN for consecutive ranges
    • Adaptive Batching: Groups scattered IDs into optimal batches
    • Slice Optimization: Converts numpy-style slicing to efficient Polars filters
    • Aggregation Expressions: Builds optimized Polars aggregation expressions
    from slaf.core.polars_optimizer import PolarsQueryOptimizer

# Build optimized filter for consecutive IDs
filter_expr = PolarsQueryOptimizer.build_optimized_polars_filter(
    list(range(100, 200)), "cell"
)

# Build submatrix filters
cell_filter, gene_filter = PolarsQueryOptimizer.build_submatrix_filter(
    cell_selector=slice(0, 100),
    gene_selector=slice(0, 5000),
    cell_count=1000,
    gene_count=20000
)

    2. SLAFArray Extensions (slaf/core/slaf.py)¶

    SLAFArray now includes Polars support alongside existing SQL functionality:

    • polars_query(expression_builder): Execute Polars queries with custom expressions
    • get_expression_lazyframe(): Get Polars LazyFrame for expression data
    • _polars_available: Property indicating Polars backend availability
    • Graceful fallback: Falls back to SQL-only mode if Polars setup fails
    # Get Polars LazyFrame
lf = slaf_array.get_expression_lazyframe()

# Apply custom operations
filtered_lf = lf.filter(pl.col("value") > 0)
result = filtered_lf.collect()

    3. Performance Metrics (slaf/core/polars_optimizer.py)¶

    The PolarsPerformanceMetrics class tracks and compares Polars vs SQL performance:

    from slaf.core.polars_optimizer import PolarsPerformanceMetrics

metrics = PolarsPerformanceMetrics()
metrics.record_polars_query("consecutive", 100, 0.1)
comparison = metrics.compare_sql_vs_polars()

    Installation¶

    Usage Examples¶

    Basic Polars Query¶

    from slaf.core.slaf import SLAFArray
import polars as pl

# Load SLAF dataset
slaf_array = SLAFArray("path/to/data.slaf")

# Get Polars LazyFrame
lf = slaf_array.get_expression_lazyframe()

# Apply operations
filtered_lf = lf.filter(pl.col("value") > 0)
aggregated_lf = filtered_lf.group_by("cell_integer_id").agg(
    pl.col("value").sum().alias("total_counts")
)

# Materialize when needed
result = aggregated_lf.collect()

    Custom Polars Query¶

    # Use polars_query for custom operations
lf = slaf_array.polars_query(
    lambda df: df.filter(pl.col("value") > 0)
                 .group_by("cell_integer_id")
                 .agg(pl.col("value").sum().alias("total_counts"))
)
result = lf.collect()

    Optimized Filtering¶

    from slaf.core.polars_optimizer import PolarsQueryOptimizer

# Build optimized filter
filter_expr = PolarsQueryOptimizer.build_optimized_polars_filter(
    list(range(100, 200)), "cell"
)

# Apply to LazyFrame
lf = slaf_array.get_expression_lazyframe()
filtered_lf = lf.filter(filter_expr)

    Performance Considerations¶

    Query Optimization¶

    The PolarsQueryOptimizer preserves the performance improvements from the SQL backend:

    • Consecutive ID Detection: Uses BETWEEN for consecutive ranges
    • Adaptive Batching: Groups scattered IDs into optimal batches
    • Range Query Optimization: Converts slices to efficient range filters
    • Boolean Mask Handling: Converts boolean masks to index lists

    Memory Efficiency¶

    • Lazy Evaluation: Operations are not executed until materialization
    • Expression Building: Complex operations built as expressions, not materialized data
    • Selective Materialization: Only materialize when explicitly requested

    Backward Compatibility¶

    All existing functionality remains unchanged:

    • SQL queries: slaf_array.query("SELECT ...") still works
    • LazyAnnData interface: No changes to user-facing API
    • LazyExpressionMatrix interface: No changes to user-facing API
    • Scanpy integration: No changes to preprocessing functions

    Error Handling¶

    The implementation includes robust error handling:

    • Graceful fallback: Falls back to SQL-only mode if Polars setup fails
    • Import error handling: Skips Polars features if not available
    • Performance monitoring: Tracks and reports performance issues
    • Comprehensive testing: Tests both success and failure scenarios

    Troubleshooting¶

    Polars Not Available¶

    If you see "Polars backend not available" errors:

    1. Install Polars: pip install polars>=0.22.0
    2. Check installation: python -c "import polars; print(polars.__version__)"
    3. Restart your Python environment

    Performance Issues¶

    If Polars queries are slower than SQL:

    1. Check query optimization: Use PolarsQueryOptimizer methods
    2. Profile queries: Use PolarsPerformanceMetrics
    3. Compare strategies: Test different optimization approaches
    4. Report issues: Include performance data in bug reports

    Memory Issues¶

    If you encounter memory issues:

    1. Use LazyFrames: Don't materialize unnecessarily
    2. Monitor memory usage: Use memory profiling tools
    3. Optimize queries: Use efficient Polars expressions
    4. Consider chunking: Process data in smaller chunks
