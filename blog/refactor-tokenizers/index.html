    Refactoring SLAF Tokenizers: From SQL to Vectorized Processing¶

    Implementation plan for achieving optimal performance and clean architecture in single-cell data tokenization

    Motivation & Current State¶

    Our recent performance optimizations revealed a fundamental architectural challenge: traditional tokenizer boundaries don't align with performance-optimized data loading.

    Current Performance Achievements¶

    • Throughput: ~260 batches/sec (8,200+ cells/sec) for both Geneformer and scGPT
    • Architecture: Lance batch-based processing with vectorized tokenization
    • Bottleneck: Device transfer overhead (resolved by device-agnostic design)

    Current Architecture Problems¶

    1. SLAFTokenizer Existential Crisis: Traditional tokenizer is bypassed in the fast path
    2. Responsibility Split: Tokenization logic split between BatchLoader (upstream) and DataLoader (downstream)
    3. Confusing Naming: "Batch" means different things in different contexts
    4. Limited Extensibility: Adding new tokenization strategies requires changes in multiple places

    Performance vs Architecture Trade-off¶

    Current Fast Path:
BatchLoader.load_batch_chunk()
  → RawBatch(gene_sequences, cell_integer_ids)
    → DataLoader._tokenize_batch_vectorized()
      → torch.Tensor

Current Slow Path:
SLAFTokenizer.tokenize_scgpt_sql()
  → SQL queries
    → list[list[int]]

    The Challenge: How do we maintain performance while providing a clean, extensible interface?

    Proposed Solution: Strategy-Based Architecture¶

    Core Philosophy¶

    1. Performance First: Vectorized tokenization stays in DataLoader
    2. Clean Separation: Each component has a single, well-defined responsibility
    3. Extensible Design: Easy to add new tokenization strategies
    4. Backward Compatibility: SQL methods remain available for legacy use

    New Architecture¶

    PrefetchBatchProcessor.load_prefetch_batch()
  → Window.apply() (scGPT/Geneformer specific)
    → Shuffle.apply() (sequential/random/stratified)
      → PrefetchBatch(gene_sequences, cell_integer_ids)
        → DataLoader._tokenize_batch_vectorized()
          → torch.Tensor

    Naming Convention¶

    Current Proposed Rationale
    BatchLoader PrefetchBatchProcessor Clear about processing prefetch batches
    RawBatch PrefetchBatch Distinguishes from training batches
    AsyncBatchPrefetcher AsyncPrefetcher Generic, agnostic about content
    WindowFunctionStrategy Window Simple, clear interface
    ShufflingStrategy Shuffle Simple, clear interface

    Terminology Clarification¶

    • Lance Fragment: Physical storage unit (~380MB each)
    • Prefetch Batch: Chunk of Arrow batches loaded by Lance's to_batches() generator
    • Training Batch: PyTorch batch for training (32 cells, padded tensors)
