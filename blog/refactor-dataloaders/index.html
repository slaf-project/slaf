    Refactoring SLAF DataLoaders: From SQL to Fragment-Based Processing¶

    Implementation plan for achieving GPU-saturating throughput in single-cell data loading

    Motivation & Opportunity¶

    Our recent experiments revealed a massive performance opportunity in single-cell data loading. Starting with SQL-based window functions that achieved only 256 cells/minute, we systematically optimized through multiple approaches:

    • SQL Window Functions: 32 cells/minute
    • Prefilter + Pandas: 35K cells/minute
    • Prefilter + Polars: 60K cells/minute
    • Fragment + Polars: 38K cells/second

    This represents a 70,000x performance improvement, transforming single-cell data loading from a bottleneck into a GPU-saturating pipeline. The key insight: Lance fragments are already optimally sized (~380MB each) and load in <0.5 seconds, making direct fragment access far more efficient than query engine-based approaches.

    With modern AI training requiring 5-6K cells/second for 8√óH100 nodes, our fragment-based approach can support ~50 GPUs directly from SSD storage, enabling the paradigm of "$100 scGPT in an hour."

    Goal: Update slaf/ml/tokenizers.py to use fragment-based processing instead of SQL queries

    Current Problem:

    • tokenizers.py now uses SQL queries via Polars (fast: 38K cells/second)
    • Our new implementation uses fragment-based Polars processing (fast: 38K cells/second)
    • This creates a performance mismatch and architectural inconsistency

    Better Architectural Approach:

    Analysis of Current Working Architecture:

    1. FragmentLoader: Tokenizer-agnostic, does Polars window functions ‚Üí returns RawFragment with gene sequences
    2. StreamingDataLoader: Takes RawFragment ‚Üí converts gene sequences to tokens ‚Üí yields batches
    3. Separation: Fragment processing (Polars) vs Tokenization (Python)

    Key Discovery: Window Functions Are Different!

    • scGPT: Uses ROW_NUMBER() and returns both gene_sequence AND expr_sequence
    • Geneformer: Uses RANK() and returns only ranked_genes
    • scGPT: Needs expression values for binning ‚Üí array_agg(value ORDER BY gene_rank) as expr_sequence
    • Geneformer: Only needs gene IDs ‚Üí array_agg(gene_id ORDER BY expression_rank) as ranked_genes

    2. Fragment Mapping Strategy¶

    def _map_cell_range_to_fragments(self, cell_range: tuple[int, int]) -> list[int]:
    """Map cell range to required fragment IDs"""
    start, end = cell_range

    # Query fragment metadata to find which fragments contain our cells
    fragment_ids = []
    for fragment_id in range(self.num_fragments):
        fragment_cells = self.get_fragment_cell_range(fragment_id)
        if self._ranges_overlap((start, end), fragment_cells):
            fragment_ids.append(fragment_id)

    return fragment_ids

    3. Polars-Based Window Functions¶

    Replace all SQL with Polars operations:

    def _tokenize_scgpt_polars(self, fragment_df: pl.DataFrame) -> pl.DataFrame:
    """Apply scGPT tokenization using Polars window functions"""

    # Rank genes by expression within each cell
    result = fragment_df.with_columns([
        pl.col("value").rank(method="dense", descending=True)
        .over("cell_integer_id")
        .alias("gene_rank")
    ]).filter(pl.col("gene_rank") <= self.max_genes)

    # Group by cell and create arrays
    final_result = result.group_by("cell_integer_id").agg([
        pl.col("gene_integer_id").alias("gene_sequence"),
        pl.col("value").alias("expr_sequence")
    ])

    return final_result

    5. Async Prefetching Strategy¶

    For 20 fragments/GPU with <1 second load times, we use threading-based prefetch queue:

    class AsyncFragmentPrefetcher:
    def __init__(self, fragment_loader: FragmentLoader, max_queue_size: int = 10):
        self.queue = Queue(maxsize=max_queue_size)
        self.worker_thread = Thread(target=self._prefetch_worker)

    def _prefetch_worker(self):
        while True:
            fragment_id = self.get_next_fragment_id()
            fragment = self.fragment_loader.load_fragment(fragment_id)
            self.queue.put(fragment)

    Why threading over asyncio:

    • Threading: Better for I/O-bound operations (disk reads)
    • Asyncio: Better for network I/O and CPU-bound tasks
    • Our case: Fragment loading is I/O-bound, so threading wins
    • Cloud note: For object store/EFS deployments, asyncio may be preferred
